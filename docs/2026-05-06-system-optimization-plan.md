# EMS 系统优化与 Agent 接入规划

> 版本: v1.0
> 日期: 2026-05-06
> 状态: Draft
> 范围: 仅规划，不涉及本轮代码改动

## 1. 文档目的

本文档用于整理 EMS 项目下一阶段的系统优化方向，覆盖以下三条主线：

1. 功能完善
2. 界面与交互优化
3. 面向外部 Agent 的可访问性建设

目标不是继续堆叠零散需求，而是把当前项目从“演示可用”推进到“稳定可用、可扩展、对 Agent 友好”的状态。

## 2. 当前状态判断

基于当前代码结构、路由、前后端页面、Agent 模块和已有文档，项目已经具备以下基础：

- 已完成设备、点检、维修、保养、备件、知识库、统计分析等核心模块基础框架
- 已具备 PC 端和 H5 端双端形态
- 已存在独立的 Agent 后端模块，包含对话、技能、知识草稿、预测分析、会话与 Artifact 存储
- 已支持飞书机器人接入

但目前整体更接近以下状态：

- 业务主链路基本成型，但仍有若干闭环缺口
- 前端体验具备演示效果，但有部分硬编码和跳转不一致问题
- Agent 能力已经有产品雏形，但更偏“嵌入式 Web 助手”，还不适合被 OpenClaw、Hermes Agent 等外部 Agent 稳定调用
- 文档中对 Agent 完成度的描述高于代码真实落地程度

## 3. 本轮评估中发现的关键问题

### 3.1 稳定性问题

- 前端单测通过，但生产构建失败
- 构建失败点为 `MobileQRScanner.vue` 依赖 `jsqr`，当前安装环境中依赖未正确解析
- 后端测试需要重新在依赖缓存稳定后完整重跑

### 3.2 路由与页面问题

- 后端存在静态路由与 `/:id` 动态路由顺序风险
- `equipment` 模块中 `/:id` 注册在 `/statistics`、`/types` 之前
- `knowledge` 模块中 `/:id` 注册在 `/search` 之前
- PC 端部分跳转目标与实际 router 定义不一致，存在无效入口风险

### 3.3 Agent 模块问题

- Agent 文档描述“L4 完成”或“100% 完成”，但代码中仍有占位实现
- `maintenance audit` 和 `analysis` 仍返回开发中占位响应
- 当前 Agent API 主要面向 Web 登录用户，缺少机器凭证、工具发现、schema 描述、流式任务协议

### 3.4 界面与产品一致性问题

- 管理助手页面有明显演示态硬编码
- 预测面板默认绑定固定设备
- 飞书配置流程中存在 `demo_appid` 等演示 fallback
- PC 与 H5 的职责边界仍不够清晰，部分页面更像“能用”，还不是“高频场景最优”

## 4. 优化总目标

下一阶段建议围绕四个结果推进：

1. 系统可稳定构建、可稳定测试、关键路由无歧义
2. 业务主流程补齐闭环，减少演示态和占位功能
3. PC 端偏管理驾驶舱，H5 端偏现场执行终端，形成清晰产品分层
4. 将 EMS Agent 从“站内 AI 页面”升级为“可被外部 Agent 稳定访问的领域能力平台”

## 5. 规划原则

- 优先修复稳定性和结构性问题，再扩功能
- 优先做高频核心场景，不在边缘功能上分散资源
- Agent 接入不直接依赖 Web 页面交互，必须有正式机器接口层
- 读写能力分级开放，写操作必须带审批或人工确认机制
- 功能、权限、证据链、审计日志必须一起设计，不能只做“能调通”

## 6. 详细规划

### 6.1 P0：稳定性与基础结构修复

#### 目标

先保证项目可构建、可测试、可访问、可继续演进。

#### 任务清单

1. 修复前端构建链路
- 校验 `jsqr` 安装与锁文件一致性
- 在干净环境执行 `npm ci`、`npm run build`
- 将构建结果纳入固定检查流程

2. 修复后端路由注册顺序
- 所有静态路径优先于 `/:id`
- 重点检查 `equipment`、`knowledge` 及其他子路由模块

3. 清理前端无效路由跳转
- 全量核对 `router.push`、`router-link` 与实际 `router/index.ts`
- 修复 PC/H5 混跳、历史遗留入口、缺失页面名

4. 建立最小可用 CI 检查
- 前端安装、构建、单测
- 后端 `go test ./...`
- Docker 启动 smoke test

5. 统一错误响应与日志基础
- 普通业务 API 与 Agent API 的错误结构尽量收敛
- 关键接口补充 trace id 或 request id

#### 验收标准

- 新环境可完整安装并通过前端构建
- 核心后端路由无歧义
- 前端主导航和关键按钮无死链
- 本地和部署环境均可通过最小 smoke test

### 6.2 P1：核心业务功能完善

#### 6.2.1 设备台账

目标：从“台账展示”升级为“完整生命周期管理”。

任务：

- 增加设备报废、封存、启用等状态流转
- 增加报废审批和审批记录
- 支持手册、图纸、附件、外部文档关联
- 增加设备责任人、专属维修工、关键财务字段的完整维护
- 增加设备历史事件时间线

#### 6.2.2 点检管理

目标：从“任务执行”升级为“可追责、可防造假”的现场点检体系。

任务：

- 强化扫码、GPS、时间窗口、照片取证
- 异常项强制备注与图片
- 点检结果自动触发后续流程时，前端明确展示联动结果
- 点检模板支持版本管理、停用/启用、复制
- 增加漏检、超时、异常点检排行

#### 6.2.3 维修管理

目标：补齐 PC 与 H5 两端维修闭环。

任务：

- 提供清晰的 PC 端报修入口
- 提供维修执行入口与状态推进界面
- 强化维修测试、确认、审核、关闭节点
- 补充维修工时、备件消耗、停机损失、维修结论
- 完善“维修转知识”流程的前后端闭环

#### 6.2.4 保养管理

目标：从“任务生成”升级为“策略型保养管理”。

任务：

- 补保养计划版本化
- 展示延期、跳过、补做原因
- 强化保养异常触发维修的闭环回显
- 增加按设备类型、工厂、责任人统计的保养效果分析
- 增加保养合规率、逾期率、重复故障关联分析

#### 6.2.5 备件管理

目标：从单库存演示态升级为企业真实可用形态。

任务：

- 支持多仓库、多库位
- 支持调拨、盘点、报损
- 增加采购建议、安全库存策略
- 增加备件批次管理
- 打通备件批次与维修故障率的关联分析

#### 6.2.6 知识库

目标：让知识库成为 Agent 与工程师共用的真实知识底座。

任务：

- 完成“维修转知识”前端功能
- 增加文档上传、手册切片、结构化搜索
- 增加知识来源、引用次数、最后引用时间
- 增加知识审核、退回、修订记录
- 支持知识关联设备类型、故障标签、处理方案

#### 6.2.7 统计分析

目标：从基础图表升级为管理层可用驾驶舱。

任务：

- 增加 MTTR、MTBF、重复故障率、逾期率、备件成本、停机损失等指标
- 增加工厂、车间、设备类型、责任人维度筛选
- 增加设备风险排行、维修成本排行、保养效果排行
- 增加面向管理层的大屏视图
- 增加“从图表跳转到明细列表”的联动能力

### 6.3 P1：界面与交互优化

#### 目标

提升系统整体一致性，让 PC 端更适合管理分析，让 H5 更适合现场作业。

#### 任务清单

1. 重构信息架构
- PC 首页定位为“管理驾驶舱”
- H5 首页定位为“我的待办 + 现场动作入口”
- 避免 PC/H5 共用不合适的页面入口

2. 优化管理助手工作台
- 左侧：会话、技能、知识、历史记录
- 中间：对话或专项分析主工作区
- 右侧：当前设备上下文、证据链、推荐动作、风险信号
- 去除演示态硬编码

3. 统一表格与列表体验
- 筛选、排序、分页、保存视图、导出
- 空状态、异常状态、重试状态统一
- 关键表格支持批量操作

4. 强化 H5 效率体验
- 扫码直达设备详情
- 底部固定核心动作入口
- 对待办、报修、执行、回填等动作减少跳转层级

5. 收敛视觉系统
- 统一色彩、圆角、阴影、按钮样式、卡片密度
- Element Plus 与 Vant 的品牌感和信息密度统一
- 减少“演示型炫效果”，强化“企业级高频操作界面”

#### 验收标准

- 主导航与高频路径不绕路
- 现场角色 3 步内完成点检/报修/保养执行
- 管理角色在首页即可发现关键风险和异常
- 设计系统在 PC/H5 两端表现一致

### 6.4 P1：面向外部 Agent 的访问能力建设

#### 目标

让 EMS 不只是内置一个聊天助手，而是成为外部 Agent 可以稳定调用的设备管理能力平台。

#### 当前问题

目前 OpenClaw、Hermes Agent 等外部 Agent 使用 EMS 的方式不够友好，原因主要包括：

- 当前接口默认基于用户 JWT 登录态
- 缺少机器身份认证方式
- 缺少工具列表、参数 schema、权限说明
- 缺少流式执行与任务状态轮询
- 缺少标准化只读/写入边界
- 缺少可直接接入 Agent 框架的协议适配层

#### 建议架构

新增独立的 `Agent Access Layer`，位于现有业务 API 和 Agent 分析层之上。

职责包括：

- 机器身份认证
- 工具发现
- 参数校验
- 权限与组织范围控制
- 执行调度
- 流式事件输出
- 审计日志

#### 建设任务

1. 机器认证
- 新增 API Key 或 OAuth Client Credentials
- 每个机器凭证绑定角色、工厂范围、可调用工具集合、速率限制
- 支持吊销、轮换、过期时间

2. 能力发现接口
- `GET /api/v1/agent/capabilities`
- `GET /api/v1/agent/tools`
- `GET /api/v1/agent/resources`

3. 任务执行协议
- `POST /api/v1/agent/runs`
- `GET /api/v1/agent/runs/:id`
- `GET /api/v1/agent/runs/:id/events`

要求：

- 支持同步短任务
- 支持异步长任务
- 支持 SSE 推送中间进度和证据

4. 工具 Schema 化
- 每个工具定义输入 JSON Schema
- 定义输出结构、错误码、权限要求
- 区分 `read_only` 与 `mutating`

5. MCP Server 适配
- 将 EMS 核心能力暴露为 MCP 工具
- 首期开放设备查询、工单查询、知识搜索、审计分析、预测分析
- 后续再开放写操作类工具

6. 写操作保护机制
- 默认外部 Agent 仅允许只读能力
- 写操作支持 `dry_run`
- 真正执行派单、建单、更新状态等动作时，要求审批令牌或人工确认

7. 证据链与审计
- 每次响应必须返回 `trace_id`
- 分析结果返回 `artifact_id`
- 返回 `evidence`、来源表、来源对象 ID、置信度
- 记录机器调用日志、用户代理、作用域、调用结果

8. 开发者接入材料
- 生成 OpenAPI 文档
- 提供 curl 示例
- 提供 Python 示例 SDK
- 提供 OpenClaw / Hermes Agent 最小接入示例

#### 首期推荐开放的 Agent 工具

1. `list_equipment`
2. `get_equipment_profile`
3. `get_equipment_prediction`
4. `list_repair_orders`
5. `get_repair_order_detail`
6. `list_maintenance_tasks`
7. `search_knowledge`
8. `run_repair_audit`
9. `run_maintenance_recommendation`
10. `get_dashboard_summary`

#### 首期不建议开放的能力

- 直接创建/关闭维修工单
- 直接修改库存
- 直接写入知识正式库
- 跨工厂读取数据
- 无审批的批量写操作

### 6.5 P2：Agent 能力深化

#### 目标

在可稳定接入的前提下，再提升 Agent 的智能性、主动性与可复用性。

#### 任务清单

1. 对话链路升级
- 从“自由对话”升级为“意图识别 -> 工具调用 -> 证据整理 -> 模型总结”

2. 技能系统版本化
- 技能草稿、审核、启用、回滚、效果统计

3. 混合检索升级
- 关键词 + 结构化过滤 + 向量检索
- 强制权限范围过滤

4. 主动洞察任务化
- 每日风险设备扫描
- 逾期保养预警
- 重复故障检测
- 备件低库存预警
- 异常维修成本预警

5. Agent 评测体系
- 建立设备管理问题集
- 验证越权风险、证据准确性、建议可执行性

## 7. 推荐实施顺序

### 第一阶段

- 构建修复
- 路由修复

## 9. 风险与依赖

### 风险

- 如果继续把 Agent 功能堆在 Web 对话页面里，会放大技术债
- 如果没有统一的机器认证与权限模型，外部 Agent 接入会很危险
- 如果知识库和检索质量不足，Agent 输出会显得“会说但不准”
- 如果 PC/H5 职责不分，后续每加一个功能都会变得更重

### 依赖

- 前后端基础构建链路稳定
- OpenAPI 或 schema 文档可持续维护
- 组织权限模型先固定
- 业务核心表结构和状态流尽量稳定后，再做大规模 Agent 接入

## 10. 最终建议

当前阶段最重要的不是继续包装“AI 助手页面”，而是把 EMS 的领域能力抽象成正式平台能力。

建议将下一阶段定位为：

`EMS vNext = 稳定的设备管理系统 + 可审计的 Agent 能力平台`

换句话说：

- 对人，EMS 是管理驾驶舱和现场作业系统
- 对 Agent，EMS 是有权限边界、有工具 schema、有证据链、有审计能力的工业领域能力底座

只有这样，OpenClaw、Hermes Agent 等外部智能体接入后，才会真正“方便访问”，而不是只能绕着 Web 页面或飞书机器人做间接调用。
