# 生产环境级别 Mock 数据全量补齐计划

## 1. 目标与数据规模
当前系统中的数据量过小，无法真实模拟工厂的实际运转情况，也无法充分发挥 Agent 的大数据分析、异常检测和趋势预测能力。本计划旨在通过程序化生成的方式，在本地 Memory 模式和 Docker 数据库部署模式下，注入生产级别的数据量。

**目标数据规模：**
- **组织人员**: 操作工(10人)、维修工(5人)、设备工程师(2人)。
- **设备台账**: 至少 100 台设备，涵盖 5 种以上不同设备类型。
- **备件库存**: 至少 20 种备件，拥有不同的初始库存和安全库存。
- **点检配置**: 每种设备类型独立配置详细的点检模板与项目。
- **保养配置**: 每台设备均有定期保养计划与内容。
- **业务流水**: 大量具有差异化执行结果的点检记录、保养记录、包含备件更换的维修工单，以及完整的备件出入库流水。

---

## 2. 实体关系与绑定梳理

为了保证业务逻辑的连贯性，生成的数据必须严格遵循实体间的关联关系：

1. **用户与设备绑定**:
   - 设备绑定责任维修工 (`DedicatedMaintenanceID`)。
   - 点检任务下发给操作工 (`AssignedTo`)。
   - 保养/维修任务下发给特定的维修工 (`AssignedTo`)。
2. **设备类型与点检/保养模板绑定**:
   - 同一类型的设备共享相同的点检模板 (`InspectionTemplate`) 和保养计划 (`MaintenancePlan`)。
3. **执行差异化关联**:
   - **点检 -> 维修**: 部分设备的点检记录随机生成 "NG" (不合格)，从而引发关联的维修单 (`RepairOrder`)。
   - **保养 -> 维修**: 部分保养记录生成 "NG"，引发关联维修单。
4. **维修与备件消耗绑定**:
   - 维修单中记录故障现象和解决方案。
   - 维修过程伴随随机数量的备件消耗 (`SparePartConsumption`)。
   - 备件消耗会联动引发库存扣减 (`SparePartInventory`) 和出库流水 (`SparePartTransaction`)。

---

## 3. 数据生成详细策略

我们将采用**程序化循环生成 + 随机微调**的策略：

### 3.1 组织人员生成
- 循环生成 10 名操作工 (`operator_1` 到 `operator_10`)。
- 循环生成 5 名维修工 (`maint_1` 到 `maint_5`)。
- 循环生成 2 名工程师 (`eng_1`, `eng_2`)。

### 3.2 备件基础数据生成
- 预定义 24 种工业常用备件名称及规格（轴承、液压泵、滤芯、变频器等）。
- 随机生成 50-500 的初始库存，并记录入库流水。

### 3.3 设备与模板生成
- 预定义 5 大类设备：`数控机床`, `全自动冲床`, `工业机器人`, `注塑机`, `自动化流水线`。
- 为每类设备编写专属的点检和保养项标准。
- 循环生成 100 台设备，按比例分配类型，并随机分配责任维修工。

### 3.4 业务历史数据生成（过去 15 天）
针对这 100 台设备，使用时间回溯循环生成历史记录：
- **点检数据**: 每天生成一次点检任务。98% 概率 OK，2% 概率项目随机 NG。
- **保养数据**: 每 15 天生成保养记录。
- **维修工单**: 当点检或保养出现 NG 时，自动生成维修工单，并随机挑选备件生成消耗记录（出库流水）。设置不同的工单状态（已关闭、维修中）。

## 4. 代码落地实施方案

由于我们需要同时兼顾 `Memory 模式` 和 `Docker 数据库`：

### 步骤 一：重构 `backend/internal/repository/seeder.go` (Docker DB 环境)
1. 使用 `rand.New(rand.NewSource(time.Now().UnixNano()))` 保证每次随机性。
2. 循环生成用户、备件和 100 台设备。
3. 循环 15 天历史，生成点检、保养和异常触发的维修单。
4. 使用 GORM 的批量插入提高效率。

### 步骤 二：重构 `backend/pkg/memory/store.go` (本地 Memory 环境)
1. 采用与 `seeder.go` 完全一致的业务生成逻辑。
2. 直接向内存中的 `map` 注入结构体对象，并手动维护外键 ID 关联。
3. 补齐 `SparePartTransactions` 的存储与初始化。
